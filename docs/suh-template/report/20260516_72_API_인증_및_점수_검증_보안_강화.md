# 🚀[기능개선][Backend] API 인증 및 점수 검증 보안 강화 (점수 검증 부분)

## 개요

랭킹에 `2,147,483,647` (INT32 MAX) 점수/콤보가 등록된 치팅 사례(2026-05-16) 대응. 치팅 레코드 2건(`id=75 김서진뚫지마라`, `id=76 ㅂㅁㅂㅁ`) DB 영구 삭제. 추가로 `POST /players/result` 엔드포인트에 score/stage/combo 상한 검증을 추가해 향후 동일 치팅 재발을 차단한다. (이슈에 명시된 `X-API-Key` 헤더 인증은 본 작업 범위 외 — 후속 PR로 진행 예정.)

## 변경 사항

### 백엔드 검증 로직
- `backend/src/apis/player_router.py`: `SaveResultRequest` Pydantic 모델에 `score`/`stage`/`combo` 상한값 추가 (`le=`). 상한 상수(`MAX_SCORE`, `MAX_STAGE`, `MAX_COMBO`)를 모듈 상단에 분리하고 산정 근거를 주석으로 명시.

### DB 직접 조작 (1회성)
- 운영 PostgreSQL(`suh-project.synology.me:5430/pickerpicker`)에 `asyncpg`로 직접 접속해 다음 SQL 실행:
  ```sql
  DELETE FROM players WHERE id IN (75, 76);
  ```
- 사용자 사전 승인 후 진행. 잔존 치팅 로우 0건 확인.

## 주요 구현 내용

### 상한 산정 근거

| 필드 | 상한 | 산정 근거 |
|------|------|-----------|
| `score` | `10,000,000` | 현재 최고 32,500. 50+ 스테이지 + 콤보 배율 확장 여유 |
| `stage` | `100` | 현재 1~15. 메이저 콘텐츠 확장 여유 |
| `combo` | `9,999` | 곡당 노트 ~300. 메들리/긴 곡 커버 가능한 4자리 |

향후 게임 콘텐츠 확장 시에도 일반 플레이 범위는 모두 통과한다. INT32 MAX(2,147,483,647) 같은 명백한 치팅 값은 모두 차단.

### 검증 동작

기존 검증:
```python
score: int = Field(..., ge=0)
stage: int = Field(..., ge=1, le=15)
combo: int = Field(..., ge=0)
```

변경 후:
```python
MAX_SCORE = 10_000_000
MAX_STAGE = 100
MAX_COMBO = 9_999

score: int = Field(..., ge=0, le=MAX_SCORE)
stage: int = Field(..., ge=1, le=MAX_STAGE)
combo: int = Field(..., ge=0, le=MAX_COMBO)
```

상한 초과 요청 시 FastAPI가 자동으로 HTTP 422 `Unprocessable Entity` 응답을 반환한다. 핸들러 진입 전 차단되므로 DB 갱신 경로 자체에 도달하지 않는다.

### `play_count` 검증 미적용 이유

`play_count`는 클라이언트 입력값이 아니라 서버 측 `save_game_result`에서 `+= 1`로 직접 증가하므로 별도 상한 검증이 불필요하다.

## 주의사항

- **이슈 #72의 잔여 작업**: `X-API-Key` 헤더 미들웨어 인증, `backend/.env` / `BACKEND_ENV_FILE` Secret에 `API_KEY` 설정, 프론트엔드 요청 헤더 자동 포함 — 본 PR 범위 외. 별도 작업으로 진행 필요.
- **현재 한계**: 점수 상한만 검증하므로 *상한 이하의 비정상 점수* (예: `9,999,999`)는 여전히 통과한다. 근본 차단은 후속 `X-API-Key` 인증 추가 시 완성된다.
- **상한 변경 시점**: 향후 콘텐츠 확장으로 정상 점수가 상한 근처에 도달하면 `MAX_SCORE` 상수만 상향 조정하면 된다.
